## 总原则

Codex 的目标是帮助用户安全、可控地完成产品 MVP 开发。效率重要，但不能以丢失文件、误删代码、覆盖配置、暴露密钥或破坏已有功能为代价。所有高风险操作必须先说明、再确认、后执行。

## 开工前 Git 保护规则

每次开始修改代码前，Codex 必须先检查当前项目是否处于 Git 管理状态，并遵守以下规则：

1. 开工前必须先检查：
   - 当前分支
   - git status
   - 是否存在未提交修改

2. 如果发现项目还没有 Git 初始化，必须先提醒用户：
   “当前项目未初始化 Git，建议先执行 git init 并提交一次备份后再继续。”

3. 如果发现存在未提交修改，必须先提醒用户：
   “当前存在未提交修改，建议先提交备份，避免后续修改难以回退。”

4. 未经用户明确确认，Codex 不允许执行：
   - `git reset --hard`
   - `git clean -fd`
   - `git checkout .`
   - 删除未提交文件
   - 强制覆盖用户已有修改

5. 如用户允许，Codex 可以建议执行以下备份命令：

   ```bash
   git add .
   git commit -m "backup before codex changes"
   ```

## 项目简介

TalentFlow AI｜HR 招聘分析 Agent 自动化工作流 MVP。

本项目是为 AI 产品经理（Vibe Coding 方向）岗位定制的招聘分析 Agent MVP，用于展示基于真实业务场景的需求拆解、产品流程设计、AI Agent 工作流设计、RAG 引用依据、AI 输出边界和快速原型落地能力。

当前版本已包含候选人管理、简历导入解析、AI 候选人评审、招聘数据看板、「招聘知识库 RAG MVP」和「招聘分析 Agent 工作台」。招聘分析 Agent 围绕候选人资料、岗位 JD 与招聘知识库，生成 JD 分析、候选人初筛、面试追问和面试后评审报告，最终由 HR 人工确认。

## 已完成核心功能

- 候选人库：新增、编辑、删除、搜索筛选、详情查看、归档、状态更新。
- 简历导入：支持 txt / docx / pdf 文本解析和基础信息抽取。
- AI 候选人评审：支持 Mock AI、默认 AI 服务代理和自定义 API Key。
- 招聘知识库 RAG MVP：支持 JD、面试标准、招聘 FAQ、岗位画像、试用期标准等资料管理、自动切片、检索问答、引用来源、置信提示和无依据兜底。
- 招聘分析 Agent：支持 JD 分析、候选人初筛、面试追问生成、面试后评审 4 个固定工作流。
- Agent 报告沉淀：保存 AgentRun / AgentReport 到 localStorage，并在候选人详情中展示历史评审记录。
- 数据看板：保留招聘漏斗、来源质量、岗位维度、周/月度分析，并新增 Agent 分析次数、推荐 / 谨慎 / 暂不推进、知识库引用次数和 HR 人工确认率等指标。

## 技术栈

- Vite
- React
- TypeScript
- localStorage
- Supabase Auth / Database
- Cloudflare Worker
- 关键词 / 简单相似度 / 文档类型加权 Hybrid RAG MVP

## 开发原则

- 以 HR / HRBP / 招聘负责人视角为主。
- 保留现有候选人库、简历导入、AI 分析、数据看板、localStorage / Supabase 逻辑，不要为了新增模块重做主链路。
- 不接真实候选人隐私数据。
- 不做复杂后端、多租户、计费体系或生产级权限系统。
- AI 分析和 RAG 问答必须保留 Mock / 本地兜底能力，预留后续真实 API 接入位置。
- API Key 不允许写死在前端代码中。
- 界面风格保持干净、可信、专业、克制，像真实公司内部 HR 系统。

## Agent 工作流设计原则

- 固定工作流优先，不做拖拽式复杂自由编排。
- 当前固定工作流包括：JD 分析、候选人初筛、面试追问生成、面试后评审。
- 每次运行 Agent 必须展示执行步骤，不允许只输出最终报告。
- AgentStep 应包含状态、输出摘要、是否使用 RAG、置信度、关键结论和引用来源。
- AgentReport 必须区分：基于候选人资料的判断、基于 JD / 知识库的依据、AI 推断与建议、需要 HR 人工确认的内容。
- 所有 Agent 输出必须标注为“辅助建议”，最终由 HR / 用人部门人工确认。
- 不允许出现 AI 执行最终录用决策、流程终止、候选人通知或跳过人工确认的逻辑。

## 招聘知识库 RAG MVP 原则

- 支持上传 / 新增 JD、面试标准、招聘 FAQ、岗位画像、试用期标准等招聘资料。
- 长文本需要自动切分为知识片段，并保存文档标题、文档类型、chunk 编号、内容摘要等可追溯信息。
- HR 提问时，优先基于命中的知识片段回答，并展示引用来源、置信提示和无依据兜底。
- 如果知识库没有足够依据，必须明确提示“知识库中未找到足够依据”，不得编造公司制度、岗位要求、候选人信息或面试结论。
- 当前检索方式是关键词 / 简单相似度 / 文档类型加权的 Hybrid 检索 MVP，不是企业级 RAG。
- 不要把当前版本包装成生产级知识库系统。后续可升级 Supabase pgvector、Embedding、权限控制、操作日志和知识库版本管理。

## 安全边界

- 不接真实候选人隐私数据作为公开默认示例。
- 不在前端、README、提交记录或 GitHub 仓库中写死真实 API Key。
- 自定义 API Key 仅可保存在当前浏览器 localStorage。
- 默认 AI 服务应通过 Cloudflare Worker 或后端代理调用。
- AI 输出不作为最终录用 / 淘汰决定。
- 无知识库依据时必须明确提示“依据不足，不能生成确定结论”。

## 后续可扩展方向

- 将 AgentRun / AgentReport 接入 Supabase Database、RLS、操作日志和面试官协作流程。
- 将招聘知识库升级为更完整的语义检索和知识库治理方案。
- 增加权限控制、操作日志、知识库版本管理、文件隐私保护和企业级安全审计。
- 支持候选人 AI 分析调用指定 JD / 面试标准作为依据。
